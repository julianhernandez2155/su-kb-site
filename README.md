# su-kb-site

Public knowledge base for **Syracuse University's ITS Data & AI department**, replacing Confluence with a GitHub Pages site styled to match [clementine.syr.edu](https://clementine.syr.edu/).

## What this is

Two consumers, one site:

1. **Humans browse** the rendered HTML at `https://<owner>.github.io/su-kb-site/` (clementine styling — SU navy/orange tokens, Sherman Sans).
2. **Claude's `WebFetch`** reads a parallel raw `.md` mirror at the same path with a `.md` suffix, plus a curated `/llms.txt` index at the site root. An org-wide Claude skill routes student/staff questions to the right pages here.

The site is generated by a **small (~450-line) Python renderer** — no SSG framework. Source pages are plain GitHub-Flavored Markdown with an 8-field YAML frontmatter; in-corpus links are ordinary relative `.md` links (not Obsidian wikilinks). Multi-department-ready: `/data-ai/` is the first namespace; other departments can follow under `/<department>/` without restructuring.

> **History:** this started on Quartz v4 (chosen for native Obsidian `[[wikilink]]` parsing). An empirical test then showed Claude's `WebFetch` reads relative-link Markdown just as well, which removed Quartz's only load-bearing justification — so it was replaced by the thin Python renderer. Rationale: [`docs/decisions/0002-pivot-from-quartz-to-thin-renderer.md`](docs/decisions/0002-pivot-from-quartz-to-thin-renderer.md).

## Architecture at a glance

```
Confluence (ITSAI space)
        │   export-tool/   (httpx + lxml; ADF/storage XML → Markdown)
        ▼
site/content/data-ai/**.md      (8-field frontmatter, relative .md links)
        │   tools/render.py   (markdown-it-py + Pygments + Jinja2 over _design/)
        ▼
site/_site/   ├── rendered HTML       (clementine styling, for humans)
              ├── raw .md mirror       (for Claude WebFetch)
              ├── llms.txt / sitemap.xml / robots.txt
              └── per-section hub pages + collapsible nav
        │   GitHub Actions  (.github/workflows/deploy.yaml)
        ▼
GitHub Pages (public)  ──  Humans browse  ·  Claude WebFetch (via skill)
```

## How it's built

- **`export-tool/`** — Python package (`su_kb_export`) that pulls a Confluence space and writes clean Markdown: ADF/storage parsing, macro handling, attachment download, the 8-field frontmatter schema, relative-link rewriting, and a test-page exclusion filter. Re-runs are idempotent via `.sync-state.json`.
- **`tools/render.py`** — the renderer. Reads `site/content/**`, renders bodies with markdown-it-py (GFM + callouts + footnotes) + Pygments, injects them into Jinja2 templates derived from `_design/`, and emits HTML + a byte-faithful `.md` mirror + `llms.txt` + `sitemap.xml` + `robots.txt`, plus per-section hub pages. Stylesheet/script URLs are content-hashed (`?v=`) for cache-busting on redeploy.
- **`_design/`** — the clementine design system: tokens, the doc-page + landing Jinja templates, and CSS.
- **`docs/`** — STATUS, ADRs, and session logs (the `decision-log` skill).

## Setup

### Prerequisites
- Python 3.11+
- `gh` CLI authenticated to GitHub
- An Atlassian API token with SU Confluence read access ([create one](https://id.atlassian.com/manage-profile/security/api-tokens))

### 1. Export content from Confluence

```powershell
git clone https://github.com/julianhernandez2155/su-kb-site.git
cd su-kb-site
Copy-Item .env.example .env
# Edit .env — fill in ATLASSIAN_EMAIL (@syr.edu) + ATLASSIAN_TOKEN

cd export-tool
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -e .
python -m pytest                       # sanity check (73 tests)
python scripts/export_space.py ITSAI   # -> ../site/content/data-ai/**.md
```

### 2. Render the site

```powershell
cd ..
pip install markdown-it-py mdit-py-plugins pygments jinja2 pyyaml
python tools/render.py                 # -> site/_site/
```

`site/_site/` is the deployable output. Pages use an absolute `/su-kb-site` base path, so opening a file directly won't load CSS — serve it under that prefix to preview locally, or just push and let CI deploy.

### 3. Deploy

Push to `main`. [`.github/workflows/deploy.yaml`](.github/workflows/deploy.yaml) runs `python tools/render.py` and publishes `site/_site/` to GitHub Pages. No pre-built output is committed; CI renders from `site/content/`.

Salvaged from the prior [`su-kb-pipeline`](https://github.com/julianhernandez2155/su-kb-pipeline) prototype after the architecture pivoted from a FastMCP + FTS5 RAG service to this static-site + Claude-skill model. Only the Confluence-export half carried over.

This site is **not** the official Clementine product, despite shared visual language.
